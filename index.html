結論

本番想定AIエージェントは「勝手に実行しない業務OS」として設計する

LLMは判断を支援し、実行はAgent OrchestratorとTool Gatewayが制御します。 社内データ検索、業務SaaS連携、承認、監査ログを分けることで、デモでも本番の安全境界を説明できます。

RAGで社内知識を検索 RBAC / ABACで権限制御 approval_requiredで人間承認 MCPでTool接続を標準化 マルチエージェントで役割分担 監査ログで追跡可能

理由

  • ✓LLMに直接業務システムを触らせず、許可されたToolだけを渡す。
  • ✓個人情報や機密情報は検索・表示・実行の前後でマスキングする。
  • ✓更新・送信・作成などの副作用は、人間承認後にだけ実行する。
  • ✓全判断とTool結果を監査ログに残し、あとから説明できる状態にする。

本番ガードレール

AIの賢さより先に、実行境界・データ保護・説明責任を固定します。

安全境界

認可: RBAC / ABAC

役職、部署、勤務地、案件担当などを使って、見られる情報と実行できる操作を分けます。

  • 一般社員は自分の申請だけ閲覧
  • 経理担当は部門横断で確認
  • 管理者のみポリシー変更

Tool境界: allowlist / blocklist

LLMへ渡すToolを毎回絞り込み、危険なToolは呼び出し候補に入れません。

  • 許可: search_policy, draft_ticket
  • 制限: send_email, update_record
  • 禁止: raw_data_export

MCP境界: Tool仕様の標準化

MCP ServerごとにTool名、入力スキーマ、認証、許可範囲を固定し、Agentが勝手に接続先を増やせないようにします。

  • Tool schemaを事前登録
  • 接続先ごとに権限分離
  • 入出力を監査ログへ保存

Agent境界: 役割分担と相互レビュー

Supervisor、Planner、Research、Action、Reviewを分け、実行担当Agentの判断をReview AgentとSupervisor Agentが確認します。

  • 単独Agentに全権限を持たせない
  • 操作前に別Agentが検査
  • Supervisorが停止・差し戻しを判断

監督: Supervisor Agent

専門Agentの出力を統合し、根拠不足、権限不足、危険操作、品質不足があれば実行を止めます。

  • 高リスク処理は人間承認へ昇格
  • Agent間の矛盾を検出
  • 最終回答の品質を確認

承認: approval_required

作成、送信、更新、削除など副作用のある操作は、人間承認を待ってから実行します。

  • 回答のみなら即時返答
  • 下書き作成は承認待ち
  • 本番更新は二重承認

データ保護: PIIマスキング

個人情報、顧客情報、給与、口座番号などは、検索結果とLLM入力の前に必要最小限へ加工します。

  • 氏名やメールを伏せ字化
  • 不要な列は渡さない
  • 出力前にも再検査

安定運用: rate limit

大量実行や異常な連続操作を止め、LLM費用と業務SaaSの負荷を制御します。

  • ユーザー単位の上限
  • Tool単位の上限
  • 異常時の自動停止

説明責任: 監査ログ

誰が、いつ、何を依頼し、どの根拠で、どのToolを実行したかを保存します。

  • プロンプトと検索根拠
  • 承認者と承認時刻
  • Tool入力と実行結果

デモシナリオ: 経費精算ルールの確認とチケット下書き

質問回答で終わる場合と、業務操作へ進む場合の違いを見せます。

社員

承認待ち

Action AgentがMCP経由でチケット下書きを作成する前に、Review AgentがPIIと権限を確認し、Supervisor Agentが最終リスクを判定してから承認者へ回します。

経費ルール検索 即時実行 閲覧権限のある社内規程をRAGで検索するだけなので副作用がありません。
